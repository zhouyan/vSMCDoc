4 # Introduction
5 
6 vSMC library provide a framework for implementing SMC algorithms. It provides a
7 core module which perform resampling, etc., operations common to all SMC
8 algorithms and applications. In addition, it provides the bases for
9 implementing parallelized samplers. The SMC algorithms are highly
10 parallelizable, but there are many frameworks for doing this. This library
11 tries to hide the different parallelization mechanism behind a unified
12 interface, and thus increases code reuse.
13 
14 # Installation
15 
16 This is a header only template C++ library. To install the library just move
17 the contents of the `include` directory into a proper place, e.g.,
18 `/usr/local/include` in Unix-alike systems. Alternatively, one can use
19 [CMake][CMake] (2.8.3 or later required),
20 ~~~sh
21 cd /path_to_vSMC_source
22 mkdir build
23 cd build
24 cmake ..
25 make install
26 ~~~
27 One may need `su` or `sudo` permissions to perform the last installation step.

55 # Parallelization backends
56 
57 The library support various backends for multi-thread parallelization, unified
58 under a uniform interface. One is C++11 concurrency. For a full C++11
59 implementation, this means no third-party dependency is required to write a
60 parallel SMC sampler. Other third-party parallelization include, [Intel Cilk
61 Plus][Intel Cilk Plus], [Intel TBB][Intel TBB] and [OpenMP][OpenMP]. [Apple
62 Grand Central Dispatch][Apple GCD] is also supported on Mac OS X and on Linux
63 via [libdispatch][libdispatch]. [Microsoft Parallel Patterns Library][MS PPL]
64 is supported on Windows when compiled with MSVC 2010 or later. In addition,
65 this library also support using [OpenCL][OpenCL] for GPGPU computing, though
66 the interface is different than others.

68 # Third-party dependencies
69 
70 This library has no dependences other than C++ standard libraries (C++11). Any
71 C++11 language features are optional.
72 
73 In particular, the library use the `<functional>` and `<random>` headers, which
74 are parts of the C++11 standard libraries. Equivalences can be found in
75 [Boost][Boost]. By default the library will use the [Boost][Boost] library as
76 C++11 implementations are not mature at the time writing. But if the C++
77 implementation has them correctly implemented, the standard headers can also be
78 used by defining suitable macros (see reference manual for details).
79 
80 Note that this library is only tested with [Boost][Boost] 1.49 or later. Also
81 not all C++11 implementations of `<functional>` and `<random>` work properly
82 even they are present.

84 # Compiler support
85 
86 This library makes heavy use of some template metaprogramming techniques. It
87 requires a standard conforming compiler. Fortunately, most commonly used
88 modern compilers, at least in C++98 mode, is able to compile the examples
89 distributed with the library.
90 
91 This library has been regularly tested with recent [Clang][Clang], [GCC][GCC]
92 and [Intel C++ Compiler][icpc], in both C++98 and C++11 modes. In particular,
93 [Clang][Clang] 3.3 and later with [libc++][libc++] and [GCC][GCC] 4.7 and later
94 support all the C++11 features used by the library very well. [Intel C++
95 Compiler][icpc] when used with [GCC][GCC] 4.7's standard library can also
96 support all the C++11 features. When it is used with [GCC][GCC] 4.8's standard
97 library, though all features are supported, some examples fail to compile when
98 complex template constructs are involved. The issues are still under
99 investigation. The current workaround is to use the [Boost][Boost] Function
100 library instead of the standard library `<functional>` (by defining the flag
101 `-DVSMC_HAS_CXX11LIB_FUNCTIONAL=0`) when using this compiler configuration in
102 C++11 mode.
103 
104 [Microsoft Visual C++][MSVC] is also supported. Version 2008 and later are able
105 to compile the examples in C++98 mode. Version 2012 and later support most of
106 the C++11 features. However, this compiler is tested less regularly.
107 
108 Other compilers such as [Open64][Open64] were previously tested in C++98 mode
109 (most of them don't support C++11 at all). Future developments will rely more
110 on C++11 features. There are likely to be new (optional) features that are
111 C++11 only. Therefore, these outdated compilers won't be tested anymore.
112 However, for the foreseeable future, all basic features should be supported by
113 a C++98 compiler.
